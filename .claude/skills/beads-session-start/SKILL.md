# beads-session-start

The first thing run in a session that touches code. Solves the "what was I doing" problem after a device switch by reading beads as the source of truth for active state.

## When to use

- Top of any conversation that touches code, before any other engineering action.
- After a long break, after switching machines, or after returning to the project from another context.
- Whenever you're unsure what's in progress.

## How to use

**Always run as a subagent.** Raw `bd` and `git` output should not enter the orchestrator's context. The subagent runs the steps below and returns a structured 3-to-5-line summary plus a list of promotable features.

## Subagent steps

1. **Pull from origin** — the `tools/git-hooks/post-merge` wrapper auto-runs `bd dolt pull`:

   ```bash
   git pull --ff-only origin main
   ```

   If non-fast-forward (machines diverged), STOP and return `{status: "diverged", message: "..."}`. Never auto-merge.

2. **Identify in-progress work**:

   ```bash
   bd list --status in_progress --json
   ```

3. **List ready and blocked work**:

   ```bash
   bd ready --limit 5 --json
   bd blocked --json   # bd 1.0.4: `blocked` does not accept --limit; consume the full list
   ```

4. **Identify the most recent epic with open children**:

   ```bash
   bd list -t epic --status open --json
   ```

   For each open epic, check whether any of its children are `in_progress` or recently active.

5. **Compose the summary** in this shape:

   - Line 1: `"You stopped on bd-N '<task-title>' inside bd-M (<epic-title>)."` (or `"No work in progress."`)
   - Line 2: `"<N> ready tasks after it."` (or list them inline if 1-3)
   - Line 3: `"<N> ready features unblocked since."` (or list them inline if 1-3)
   - Line 4-5 (if applicable): any tasks with `BLOCKED:` comments awaiting human input. List the issue ID and the blocker question.

6. **Return a structured response** to the orchestrator:

   ```json
   {
     "status": "ok",
     "summary": "<the 3-5 line summary above>",
     "in_progress_task": "bd-N or null",
     "in_progress_epic": "bd-M or null",
     "ready_features": [
       {"id": "bd-X", "title": "Library inspector drawer"},
       {"id": "bd-Y", "title": "Building tooltip"}
     ],
     "blocked_awaiting_human": [
       {"id": "bd-Z", "question": "Should the drawer push the map left or overlay it?"}
     ]
   }
   ```

## Promotion sub-action

The orchestrator (main conversation) decides whether to offer promotion. If the user picks a feature, dispatch the promotion as a separate subagent invocation with these steps:

1. Compute slug and date:

   ```bash
   FEATURE_TITLE="<feature title from bd-X>"
   FEATURE_ID="bd-X"
   TODAY=$(date +%Y-%m-%d)
   SLUG=$(echo "$FEATURE_TITLE" | tr '[:upper:]' '[:lower:]' | tr -cs 'a-z0-9' '-' | sed 's/^-//;s/-$//')
   PLAN_PATH="docs/superpowers/plans/${TODAY}-${SLUG}.md"
   ```

2. Create the epic:

   ```bash
   EPIC_ID=$(bd create -t epic --title "$FEATURE_TITLE" --body "Plan: $PLAN_PATH" --json | jq -r '.id')
   ```

3. Create the scoping task:

   ```bash
   SCOPING_ID=$(bd create -t task --parent "$EPIC_ID" --label scoping --title "Scope and plan: $FEATURE_TITLE" --json | jq -r '.id')
   ```

4. Add the dependency edge so closing the epic closes the feature:

   ```bash
   bd dep add "$FEATURE_ID" "$EPIC_ID"
   ```

5. Write the stub plan file (use `Write` tool, not bash heredoc, to avoid escaping issues):

   ```markdown
   # <FEATURE_TITLE> Implementation Plan

   <!-- BEADS — generated by beads-session-start (promotion); do not hand-edit
   feature: <FEATURE_ID>
   epic:    <EPIC_ID>
   scoping_task: <SCOPING_ID>
   tasks:
     (none yet — will be filled in by beads-mirror-plan after writing-plans runs)
   -->

   **Plan mode:** sequential

   **Goal:** (TBD — fill in via superpowers:brainstorming → writing-plans)
   ```

6. Return:

   ```json
   {
     "status": "promoted",
     "feature": "<FEATURE_ID>",
     "epic": "<EPIC_ID>",
     "scoping_task": "<SCOPING_ID>",
     "plan_path": "<PLAN_PATH>"
   }
   ```

   The orchestrator then invites the user to launch `superpowers:brainstorming` against the stub plan path.

## Constraints

- **Read-only on beads except via promotion.** All other steps are queries.
- **The git pull is the only side effect of merely invoking this skill** (the post-merge wrapper auto-runs `bd dolt pull` as part of that pull).
- **Always run as a subagent.** Do not run these commands in the main conversation.
